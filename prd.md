# PRD (Product Requirements Document)

## 1. 문서 정보
- 문서명: 빛공해 감지를 위한 딥러닝 프로젝트 PRD
- 문서 버전: v1.0 (Draft)
- 작성일: 2026-03-06
- 작성 목적: 빛공해 자동 감지/분석 모델 및 운영 시스템 개발 요구사항 정의

---

## 2. 제품 개요

### 2.1 제품명 (가칭)
**LumiWatch AI**

### 2.2 한 줄 설명
도시 및 생활권의 야간 이미지/센서 데이터를 기반으로 빛공해 수준을 자동 감지하고, 원인 광원을 식별하여 개선 우선순위를 제시하는 딥러닝 기반 모니터링 시스템.

### 2.3 배경
- 빛공해는 생태계 교란, 수면장애, 에너지 낭비, 주민 민원 증가를 유발함.
- 기존 측정 방식(현장 계측 중심)은 시간/공간 커버리지가 낮고 운영비가 높음.
- 지자체/연구기관이 상시 모니터링 가능한 데이터 기반 의사결정 도구를 필요로 함.

---

## 3. 문제 정의
1. 현재 빛공해 분석은 수동 조사 비중이 높아 신속 대응이 어렵다.
2. 광원 유형(가로등, 광고판, 건물 외벽조명 등)별 영향 분석이 체계화되어 있지 않다.
3. 지역별/시간대별 빛공해 변화를 정량적으로 추적하기 어렵다.

---

## 4. 목표 및 성공 기준

### 4.1 비즈니스 목표
- 빛공해 민원 대응 시간 단축
- 개선 대상 구역 우선순위 자동 산정
- 야간 조명 정책 수립을 위한 근거 데이터 제공

### 4.2 제품 목표 (MVP)
- 야간 이미지 기반 빛공해 등급 분류(5단계)
- 과도광원 위치 탐지(객체 탐지)
- 구역 단위 빛공해 점수(0~100) 산출

### 4.3 핵심 성과 지표 (KPI)
- 분류 성능: Macro F1 ≥ 0.85
- 광원 탐지 성능: mAP@50 ≥ 0.75
- 점수 예측 성능: MAE ≤ 8.0
- 추론 지연: 단일 이미지 기준 평균 ≤ 300ms (T4급 GPU 환경)
- 운영 안정성: 월간 API 가용성 ≥ 99.5%

---

## 5. 사용자 및 이해관계자

### 5.1 주요 사용자
- 지자체 환경/도시관리 담당자
- 공공조명 운영 담당자
- 환경 연구기관/대학 연구자

### 5.2 이해관계자
- 정책결정자(예산/규제 부서)
- 현장 점검 인력
- 시스템 운영/개발팀

### 5.3 사용자 시나리오
1. 담당자는 전일 야간 데이터 기반 위험 구역 Top N을 확인한다.
2. 각 구역에서 과도광원 위치와 유형을 확인한다.
3. 개선 전/후 점수 변화를 비교해 정책 효과를 검증한다.

---

## 6. 범위 정의

### 6.1 포함 범위 (In Scope)
- 고정형 카메라 야간 이미지 + 공개 야간조도 데이터(위성 등) 수집/정제
- 이미지 단위 빛공해 등급 분류 모델 개발
- 광원 객체 탐지 모델 개발
- 추론 API 및 배치 분석 파이프라인 구축
- 결과 시각화를 위한 기본 대시보드용 데이터 제공 API

### 6.2 제외 범위 (Out of Scope)
- 조명 장치 자동 제어(디밍, 스케줄링) 기능
- 신규 IoT 하드웨어 설계/양산
- 법적 행정처분 자동화

## 8. 비기능 요구사항

### 8.1 성능
- API P95 응답시간: 1.5초 이하 (배치 제외)
- 동시 요청 50 rps에서 오류율 < 1%

### 8.2 안정성/운영성
- 장애 복구 시간(RTO): 2시간 이내
- 일일 백업 및 30일 보관

### 8.3 보안/개인정보
- 이미지 내 개인식별 가능 정보(얼굴/차량번호) 비식별 처리 후 저장
- 전송 구간 TLS 적용, 저장 데이터 접근 제어(RBAC)

### 8.4 확장성
- 도시 단위(최소 10개 구역) 동시 운영 가능
- 모델/데이터 버전 확장 시 무중단 배포 전략 적용

---

## 9. 데이터 요구사항

### 9.1 데이터 소스
- 고정형 CCTV/관측 카메라 야간 프레임
- 공개 야간조도 데이터(예: 위성 야간광)
- 보조 메타데이터(시간대, 날씨, 위치, 행사 일정 등)

### 9.2 데이터 규모 (MVP 목표)
- 원천 이미지: 50,000장 이상
- 라벨링 완료 이미지: 15,000장 이상
- 탐지용 bbox 라벨: 40,000개 이상

### 9.3 라벨 스키마
- 분류 라벨: 0(매우낮음)~4(매우높음)
- 탐지 라벨: `street_light`, `billboard`, `facade_light`, `sports_light`, `vehicle_light`
- 메타 라벨: 촬영환경(맑음/비/안개), 시간대, 지역코드

### 9.4 데이터 품질 기준
- 라벨 일치도(annotator agreement): Cohen’s Kappa ≥ 0.75
- 결측/손상 이미지 비율 ≤ 2%
- 학습/검증/테스트는 시간·지역 기준으로 분리하여 누수 방지

---

## 10. 모델 요구사항 및 기술 접근

### 10.1 모델 구조 (MVP)
- 분류 모델: EfficientNet 계열 백본 기반 분류기
- 탐지 모델: YOLO 계열 탐지기
- 점수 산출: 분류/탐지 결과와 메타데이터를 결합한 회귀 헤드 또는 후처리 규칙

### 10.2 학습 전략
- 데이터 증강: 밝기/노이즈/날씨 시뮬레이션, 기하학 변환
- 클래스 불균형 대응: class weight, focal loss 등
- k-fold 또는 지역 기반 hold-out으로 일반화 성능 검증

### 10.3 배포 전략
- 모델 아티팩트 버전 관리(예: MLflow + Object Storage)
- API 서빙(예: FastAPI + Torch/TensorRT)
- 배치 추론(야간 데이터 일괄 처리)

---

## 11. 평가 계획

### 11.1 오프라인 평가
- 분류: Precision, Recall, Macro F1, Confusion Matrix
- 탐지: mAP@50, mAP@50:95
- 점수: MAE, RMSE, 상관계수

### 11.2 온라인/파일럿 평가
- 현장 점검 결과와 모델 결과 일치율 비교
- 고위험 구역 선별 정확도(Top N precision)
- 사용자 만족도(담당자 설문) 5점 만점 기준 4.0 이상

### 11.3 수용(Release) 기준
- KPI 80% 이상 달성 시 파일럿 배포
- 보안/성능 필수 요구사항 충족 시 운영 전환

---

## 12. MLOps 및 운영 요구사항
- 실험 추적: 실험 파라미터/지표/모델 자동 기록
- 데이터/모델 버저닝: 재현 가능 학습 파이프라인 보장
- 모니터링: 성능 저하, 입력 분포 변화, 오류율 알림
- 재학습 정책: 월 1회 정기 + 드리프트 이벤트 기반 수시 재학습

## 13. 일정 (MVP 기준 12주)

| 기간 | 마일스톤 | 산출물 |

| 11주 | 파일럿 테스트/현장 검증 | 파일럿 결과 리포트 |
